DataClean OpenEnv — What We Built & Why

Team explainer — no jargon, plain English

1. What is this hackathon actually asking us to build?

Meta and Hugging Face created a standard called OpenEnv. Think of it like a rulebook for building AI training gyms. The hackathon says: "Build a gym for an AI agent to practice a real-world task in."

The gym must:

  • Let an AI agent take actions step by step
  • Give the agent a score after each action (reward)
  • Have at least 3 tasks — easy, medium, hard
  • Run inside Docker and deploy on Hugging Face

We chose the task: Data Cleaning. Every company that works with data spends hours fixing messy spreadsheets. We built an environment where an AI can learn to do that automatically.

2. What does "data cleaning" mean here?

Imagine you receive a messy Excel file from a client. It might have:

  • Missing values — some cells are blank or say "N/A"
  • Wrong data types — age stored as text "25" instead of number 25
  • Duplicate rows — same record appears twice
  • Inconsistent text — "USA", "usa", "U.S.A" all meaning the same country
  • Extreme outliers — amount = $500,000 when everything else is under $2,000
  • Multiple tables that need to be joined together

Our environment presents these messy datasets to an AI agent and asks it to fix them step by step.

3. How does the AI agent interact? (The Loop)

1. reset()    →  Agent receives a dirty dataset + description of what to fix
                 Score starts at 0.0

2. step()     →  Agent sends ONE action (e.g. "fill nulls in age column with median")
                 Environment applies it, scores the result
                 Agent receives: updated dataset + reward + current score

3. repeat     →  Agent keeps sending actions until it calls "submit" or runs out of steps

4. submit     →  Final score is recorded. Episode ends.

The agent never sees the "correct answer" — it only sees the current messy data and must figure out what to fix based on the feedback it gets after each action.

4. Our 4 Tasks

TaskDifficultyWhat the agent must doMax Steps
task1
Null Fixer		 Easy 50-row customer table. Fill missing age values with median → convert to integer. Fill missing salary values with mean. 10
task2
Schema Normalizer		 Medium 200-row orders table. Remove ~30 duplicate rows. Fix "usa/USA/U.S.A" → all uppercase. Convert mixed date formats to proper datetime. Fill missing amounts. 20
task3
ETL Pipeline		 Hard Two separate tables (orders + customers). Must: join them on customer_id, remove outliers in the amount column, add a new "order_year" column derived from the date. 30
task4
Data Drift ⭐		 Expert (Novel) Live streaming table. Starts with 120 dirty rows. Every 5 steps, 7 MORE dirty rows are automatically injected — simulating a real Kafka data pipeline. Agent must keep cleaning while new dirty data keeps arriving. 40
Why Task 4 is our competitive edge: No existing OpenEnv environment does this. All other environments have a fixed dataset — static, unchanging. Task 4 simulates what actually happens in production data pipelines where data keeps flowing in. This is genuinely novel.

6. Scoring (Reward System)

After every single step, the agent gets a reward:

reward = new_score - previous_score

This means:

  • If the action improved the data → positive reward (+)
  • If the action made things worse → negative reward (−)
  • If the action failed/invalid → small penalty (−0.02)

Each task has its own grader that scores specific things. For example, Task 1 grader checks:

  • 30% — is age integer type?
  • 35% — are all age nulls filled?
  • 15% — is salary numeric?
  • 20% — are all salary nulls filled?

This "partial credit" design means the agent always has a signal to improve — it's never just 0 or 1 with nothing in between.

7. What is the "Baseline"?

The hackathon requires us to show that an actual AI model can run against our environment and produce scores. Our baseline uses a free LLM (Groq's llama-3.3-70b) to play through all 4 tasks.

Our innovation: All 4 tasks run simultaneously using Python threads (parallel execution). Normal submissions run tasks one by one. Ours runs all 4 at the same time — 3x faster.

8. How the code is structured

FileWhat it does (plain English)
server/dataset_factory.pyGenerates dirty datasets. Uses a "seed" number so the same dataset is always generated the same way — ensures reproducibility. Also generates "drift batches" (new dirty rows) for Task 4.
server/graders.pyThe scoring functions. Takes the current state of the data and returns a number between 0.0 and 1.0. Completely deterministic — same input always gives same score.
server/environment.pyThe "game engine". Handles reset(), step(), and state(). For Task 4 it automatically injects drift rows every 5 steps.
server/app.pyThe web server (FastAPI). Exposes all endpoints as HTTP routes so any agent can talk to it.
models.pyDefines exactly what an Action and Observation look like using Pydantic. This is the "contract" between agent and environment.
baseline.pyRuns an LLM agent against all 4 tasks simultaneously and prints scores.
DockerfilePackages everything into a container. Runs on port 7860 (required by Hugging Face Spaces).
